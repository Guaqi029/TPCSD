1. 立即查看 per-class metrics（最重要！）
打印或记录每个 epoch 的 per-class accuracy / recall / F1（尤其是 tail 类）。
如果头部类（样本多的）acc > 80–90%，而少数 tail 类 acc < 20–30%，甚至接近 0，那就是典型的分类器偏置（classifier bias） 或虚拟采样失效。
怎么做：在 _evaluate_classifier 函数中增加 return_per_class=True，然后输出 tail 类（样本最少的 2–3 类）的指标。
这步通常 5 分钟就能看出问题所在。

2. 检查虚拟样本生成质量（第二重要）
看每个 epoch 的虚拟样本数量分配（AAS 是否真的把更多样本给尾部？）。
随机抽几个虚拟特征，计算它们与对应 class prototype 的余弦相似度（应该较高）和与真实特征的分布差异。
如果虚拟特征与 prototype 相似度很低，或方差太小（模式坍缩），说明 prototype-guided 采样没起作用。
快速检查：打印 class_sizes（虚拟数量）和采样后的虚拟特征均值/方差 vs 真实特征。

3. 查看训练/验证 loss 和 acc 的曲线（包括 per-group）
画出 head / medium / tail 三组的 acc/BAC 曲线（你代码里有 _build_count_groups 和 _group_metric_from_per_class）。
如果 head acc 很高但 tail BAC 持续低 + 波动，说明 Stage 2 补偿尾部失败。
loss 高位震荡通常意味着分类器在真实+虚拟混合数据上过拟合或欠拟合。

4. 检查 Stage 1 输出质量（快速验证）
用 Stage 1 保存的 ResNet encoder + 线性头直接在 val/test 上跑一次评估（不跑 Stage 2）。
如果 Stage 1 本身的 val_bac 已经只有 0.50 左右，那 Stage 2 再怎么补偿也难救（说明问题出在 Stage 1 的表征质量或 TPCSD 训练不足）。
如果 Stage 1 还不错，但 Stage 2 掉得很厉害，则问题在虚拟采样或 decoupled 训练。

5. 其他快速检查（10 分钟内）
学习率是否太高？（分类器 lr=1e-3 在 decoupled 阶段经常偏高，建议试 5e-4 或 1e-4 + scheduler）。
class re-weighting 是否开启？权重计算是否合理？
prototype guidance 强度 $  \gamma  $ 是否太小/太大？
虚拟特征是否 merge_real=True？如果只用虚拟样本训练，容易不稳定。
